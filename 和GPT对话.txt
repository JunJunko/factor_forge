👉 一次实验 = 多组结果，而不是一次跑一个结果

## 一、最小因子定义

这是每个因子都必须填写的部分。

```yaml
factor:
  name: sector_rank_slope
  label: 行业内强弱排名斜率
  description: 衡量股票在所属行业内相对强弱排名的变化速度
  direction: positive
  formula: slope(cs_rank(ret_5d, by=industry), window=5)
```

这几个字段的含义：

| 字段            | 含义          |
| ------------- | ----------- |
| `name`        | 唯一英文标识，不能重复 |
| `label`       | 中文名称        |
| `description` | 因子的经济含义     |
| `direction`   | 正向还是反向      |
| `formula`     | 实际计算公式      |

其中最核心的是：

```yaml
formula: slope(cs_rank(ret_5d, by=industry), window=5)
```

以后新增因子，理想状态就是只改这个公式和说明。

# 二、数据依赖

因子必须显式声明需要什么数据。

```yaml
data:
  frequency: daily
  required_fields:
    - close
    - industry
  lookback_days: 10
```

这里解决三个问题。

## 1. 数据频率

```yaml
frequency: daily
```

以后可以扩展：

```yaml
frequency: minute_5
frequency: tick
frequency: quarterly
```

但第一版最好只支持日频。

## 2. 必需字段

例如：

```yaml
required_fields:
  - close
  - volume
  - industry
```

系统在运行前先检查字段是否齐全，而不是跑到中间才报错。

## 3. 最大回看窗口

```yaml
lookback_days: 10
```

这个非常重要。

系统需要知道：

* 从哪一天开始才能产生有效因子；
* 缓存需要加载多少历史数据；
* 是否存在数据不足；
* 是否有未来数据风险。

---

# 三、计算范围

横截面因子必须说明“在哪个范围内计算”。

```yaml
scope:
  cross_section: industry
  universe: default
  min_group_size: 10
```

例如：

```yaml
cross_section: market
```

表示全市场排名。

```yaml
cross_section: industry
```

表示行业内部排名。

```yaml
cross_section: concept
```

表示概念板块内部排名。

你刚才说的行业扩散、横截面斜率，就必须有这个字段。

否则下面两个公式看起来一样，含义完全不同：

```text
全市场排名斜率
行业内排名斜率
```

# 四、输出约束

所有因子最终必须输出统一格式：

```text
trade_date
ts_code
factor_value
```

可选附加字段：

```text
factor_valid
invalid_reason
group_code
```

例如：

| trade_date | ts_code   | factor_value | factor_valid | group_code |
| ---------- | --------- | -----------: | ------------ | ---------- |
| 2026-06-01 | 000001.SZ |         0.73 | true         | 银行         |
| 2026-06-01 | 000002.SZ |        -0.21 | true         | 房地产        |

因子系统内部不管公式多复杂，最终都必须压缩成这一张标准表。

这是整个批量系统能复用的核心。

# 七、公式语言需要支持哪些基础操作

你不需要一开始支持无限复杂的表达式。

第一版支持六类算子就够了。

## 时间序列算子

```text
ret(close, 5)
mean(volume, 20)
std(close, 20)
slope(close, 10)
delta(close, 5)
max(close, 20)
min(close, 20)
```

## 横截面算子

```text
cs_rank(x)
cs_zscore(x)
cs_percentile(x)
```

## 分组横截面算子

```text
cs_rank(x, by=industry)
cs_zscore(x, by=industry)
```

## 基础数学算子

```text
+
-
*
/
abs
log
clip
```

## 条件算子

```text
where(condition, value_if_true, value_if_false)
```

## 滞后算子

```text
lag(x, 1)
```

有了这些，就已经能描述大量因子。

```yaml
formula: delta(cs_rank(ret(close, 5), by=industry), 5)
```

# 九、一个重要问题：允许公式引用中间因子吗

我建议允许。

例如先定义：

```yaml
features:
  ret_5d: ret(close, 5)
  industry_rank: cs_rank(ret_5d, by=industry)
```

然后主公式：

```yaml
formula: slope(industry_rank, window=5)
```

完整形式：

```yaml
calculation:
  features:
    ret_5d: ret(close, 5)
    industry_rank: cs_rank(ret_5d, by=industry)
  formula: slope(industry_rank, window=5)
```

这样比把所有东西塞进一行好很多：

```text
slope(cs_rank(ret(close, 5), by=industry), window=5)
```

简单因子可以写一行，复杂因子允许拆中间变量。

# 十、数据来源不要写死在因子里

因子只声明字段：

```yaml
required_fields:
  - close
  - industry
  - total_mv
```

不要写：

```yaml
close_source: tushare.daily
industry_source: tushare.index_member
```

数据源映射应该由数据层管理。

例如系统统一配置：

```yaml
field_mapping:
  close: tushare.daily.close
  industry: tushare.stock_basic.sw_industry
  total_mv: tushare.daily_basic.total_mv
```

这样以后换数据源，因子定义不用改。

---

# 十一、方向字段需要怎么定义

建议只允许三种值：

```yaml
direction: positive
direction: negative
direction: unknown
```

含义：

* `positive`：因子越大，预期收益越高；
* `negative`：因子越小，预期收益越高；
* `unknown`：让系统双向测试。

但要注意：

> `unknown` 只能用于探索阶段，进入正式样本外测试前必须固定方向。

否则系统看到哪边赚钱就选哪边，会产生选择偏差。

# 十三、这个格式是否合格，只看三个标准

新增因子时：

1. 不需要改回测代码；
2. 不需要改数据处理主流程；
3. 只新增一个YAML配置，就能生成标准因子表。

满足这三个条件，才算真正可复用。

我倾向于第一版采用 **“YAML元数据 + 受限公式DSL + 可选中间变量”**，而不是允许因子直接提交任意Python代码。任意Python更自由，但很快又会退化成一堆一次性脚本。

好，实验展开规则要解决的是：

> 一个因子提交后，系统自动生成哪些实验组合；哪些是固定必跑；哪些只在前一层通过后再跑。

这里最容易犯的错误，是把所有参数做笛卡尔积。
例如 4 个 TopN × 4 个持有期 × 3 个股票池 × 3 个中性化 × 3 个成本 × 5 个 Regime，立刻变成 2,160 组实验。效率反而更低，还会制造严重的数据挖掘偏差。

所以我建议采用：

> **固定主实验 + 分层展开 + 条件晋级**

---

# 一、先明确实验配置和因子配置的边界

因子配置只描述：

```text
这个因子怎么算
```

实验配置描述：

```text
在哪些条件下验证这个因子
```

同一个因子配置应能被多个实验模板复用。

例如：

```yaml
factor_ref: sector_rank_slope
experiment_profile: short_term_cross_section_v1
```

因子不需要知道：

* Top2还是Top20
* 持有1天还是10天
* 成本多少
* 用什么基准
* 测试哪个时间区间

---

# 二、不要把所有实验放在同一层

我建议将实验分成四层。

```text
L0 数据与因子质量检查
↓
L1 预测能力筛选
↓
L2 可交易性回测
↓
L3 稳健性与归因
```

只有上一层满足最低要求，才进入下一层。

这样一个明显无效的因子，不需要跑完整交易回测和几十种稳健性测试。

---

# 三、L0：数据与因子质量检查

这一层不是验证 Alpha，而是确认因子算得对。

每个因子必须执行，不产生大量参数组合。

## 固定检查项

```yaml
quality_checks:
  - coverage
  - missing_rate
  - unique_value_ratio
  - cross_section_count
  - distribution
  - extreme_value_ratio
  - temporal_stability
  - future_data_audit
```

## 建议输出

* 每日覆盖股票数
* 缺失率
* 因子均值、标准差、分位数
* 因子值是否长期不变
* 每日有效横截面数量
* 分组样本是否足够
* 因子最大回看期
* 是否引用未来字段
* 是否在公告实际可得日之前使用数据
* 因子值是否有 `inf`、异常极值

## 自动失败条件

例如：

```yaml
l0_gate:
  min_coverage: 0.70
  max_missing_rate: 0.30
  min_daily_cross_section: 100
  min_unique_ratio: 0.01
  future_data_violation: 0
```

行业内因子还要额外检查：

```yaml
min_group_size: 10
min_valid_groups_per_day: 5
```

L0失败就停止，不进入IC和回测。

---

# 四、L1：预测能力筛选

这一层主要看：

> 因子与未来收益之间是否存在稳定关系。

不急着模拟真实交易。

## 1. 固定预测周期

结合你持仓1至15天的风格，我建议第一版固定：

```yaml
forward_horizons:
  - 1
  - 3
  - 5
  - 10
  - 15
```

这里的含义是未来累计收益，不是一定持有这么久。

1日看极短期反应，3至5日看核心短线效果，10至15日看衰减和反转。

不建议第一版再加20、30、60日。与你的实际交易风格关系不大。

---

## 2. 固定因子版本

第一版建议只展开两个版本：

```yaml
variants:
  - raw
  - neutralized
```

其中 `neutralized` 建议固定为：

```yaml
neutralize_by:
  - industry
  - log_market_cap
```

但这里有个重要原则：

> 行业内横截面因子，不一定需要再次做行业中性化。

例如：

```text
cs_rank(ret_5d, by=industry)
```

本身已经在行业内排名。

所以展开规则应该根据因子作用域自动判断：

```yaml
variant_rules:
  market_cross_section:
    - raw
    - industry_size_neutralized

  industry_cross_section:
    - raw
    - size_neutralized
```

否则会重复中性化。

---

## 3. 固定股票池

第一版不要同时测试过多股票池。

建议固定两个：

```yaml
universes:
  - tradeable
  - liquid
```

### tradeable

基本可交易股票池：

* 排除ST
* 排除退市整理
* 排除停牌
* 排除上市未满一定天数
* 排除无有效价格数据
* 保留较广泛的市场覆盖

### liquid

在 `tradeable` 基础上增加流动性门槛：

* 过去20日平均成交额门槛
* 或全市场成交额分位数过滤
* 排除极低价股票
* 排除成交不连续股票

我不建议用：

```text
full
candidate
```

作为通用名字。

`full` 容易让人误以为包含不可交易股票，`candidate` 又不清楚筛选逻辑。股票池名称应该表达真实规则。

---

## 4. L1固定输出

每个组合：

```text
因子版本 × 股票池 × 预测周期
```

输出：

* Pearson IC
* Rank IC
* IC均值
* IC标准差
* ICIR
* IC正值比例
* t值
* p值
* FDR q值
* 分组收益
* Top-Bottom收益
* 单调性
* IC衰减曲线
* 月度IC
* 年度IC
* OOS IC

---

## 5. L1不要做Top2

因为L1是验证普遍预测关系。

Top2样本太少、噪声太大，应该放到可交易性层。

L1分组建议：

```yaml
quantile_groups: 10
```

当每日股票数不足时自动降为5组。

---

# 五、L2：可交易性回测

只有L1发现预测线索后，才进入真实组合测试。

但这里不能只允许IC强的因子晋级。

因为你的策略可能属于：

> 全截面IC不高，但极端头部有效。

所以L1晋级规则应该允许两条路径：

```text
路径A：全截面有效
路径B：头部收益异常明显
```

---

## 1. TopN展开规则

结合你实际每天选Top2，我建议：

```yaml
top_n:
  - 2
  - 5
  - 10
  - 20
```

不需要默认跑Top1。

Top1过于集中，极容易被单只股票、涨停板和不可成交问题主导。可以作为诊断项，但不作为主实验。

这四个TopN的作用不同：

| TopN | 用途            |
| ---: | ------------- |
|    2 | 对应你的实际策略      |
|    5 | 检查头部收益是否可扩展   |
|   10 | 检查信号是否只有极少数有效 |
|   20 | 检查容量和尾部衰减     |

---

## 2. 持有期与调仓频率必须分开

这是非常重要的设计。

很多一次性回测会把：

```text
持有5天
```

和：

```text
每5天调仓
```

混成一件事。

实际上应该分开：

```yaml
rebalance_frequency:
  - 1D

holding_periods:
  - 1
  - 3
  - 5
  - 10
  - 15
```

默认每天产生新信号。

对于持有期大于1天，可以使用重叠组合：

* 每天新建一批仓位
* 每批持有固定天数
* 总组合由多个信号批次叠加

这样更适合评估因子的真实收益衰减。

第一版不建议同时展开：

* 每日调仓
* 每周调仓
* 每月调仓

否则实验空间会迅速变大。

---

## 3. 买入价格规则必须固定

第一版建议只选一个主口径，不要同时优化多个成交价。

推荐：

```yaml
signal_time: close_t
entry_price: open_t1
exit_price: open_after_holding_period
```

含义：

* T日收盘后计算因子
* T+1日开盘买入
* 持有若干交易日
* 到期日开盘卖出

这个口径相对保守，也更容易避免未来数据。

可以保留一个诊断版本：

```yaml
entry_price_diagnostic: close_t1
```

但不能拿更好看的成交价作为主结果。

---

## 4. 权重规则

第一版固定等权：

```yaml
weighting: equal
```

不要一开始加入：

* 因子值加权
* 风险平价
* 波动率倒数
* 优化器权重

那些属于后续策略层，不属于因子Alpha初筛。

---

## 5. 交易约束

必须统一固定：

```yaml
execution_constraints:
  exclude_suspended: true
  cannot_buy_limit_up: true
  cannot_sell_limit_down: true
  min_listing_days: 60
  exclude_st: true
  exclude_delisting_period: true
```

还要明确涨跌停判断使用什么价格口径和交易制度。

如果实际做A股，这些不能省略。

---

## 6. 成本展开规则

不要用太多成本档位。

建议：

```yaml
cost_scenarios_bps:
  - 0
  - 10
  - 20
```

含义：

* 0bps：观察毛Alpha
* 10bps：相对乐观
* 20bps：主评估口径

如果你的20bps已经包含双边全部成本，就要在系统里明确，不要含糊。

建议拆字段：

```yaml
cost_model:
  commission_bps_per_side: 3
  slippage_bps_per_side: 5
  stamp_duty_bps_sell: 5
```

然后系统计算真实总成本。

比直接写一个20bps更清楚。

---

## 7. 基准展开

建议固定三类基准：

```yaml
benchmarks:
  - universe_equal_weight
  - market_index
  - style_matched
```

### universe_equal_weight

同一股票池每日等权。

这是最重要的基准，因为可以判断：

> 策略是否真的有选股能力，而不是股票池本身上涨。

### market_index

例如中证全指、中证1000或与你策略风格相近的指数。

### style_matched

市值、行业暴露匹配后的组合。

第一版可以先输出风险归因，不一定要做完整可交易的风格匹配组合。

---

# 六、L3：稳健性与归因

不是所有因子都跑。

只有L2有成本后正收益或明确头部异常，才进入这一层。

## 固定展开项

```yaml
robustness_tests:
  - year_by_year
  - rolling_window
  - train_validation_test
  - topn_curve
  - holding_decay
  - cost_sensitivity
  - contribution_concentration
  - industry_exposure
  - size_exposure
  - removal_of_best_trades
```

---

## 1. 时间切分

建议系统固定采用两种结果。

### 全样本描述

用于理解因子。

### 样本外验证

用于判断是否可靠。

第一版可以采用滚动切分：

```yaml
walk_forward:
  train_years: 3
  validation_years: 1
  test_years: 1
  step_years: 1
```

但是因子公式没有训练参数时，`train` 的作用主要是：

* 确定方向
* 确定合理窗口族
* 确定是否晋级

最终测试集不能再根据结果修改因子。

---

## 2. 参数邻域稳定性

系统不能只测试用户提交的单点参数。

但也不能无边界搜索。

建议因子输入中标识哪些参数允许做邻域测试，例如：

```yaml
calculation:
  parameters:
    return_window:
      value: 5
      robustness_neighbors: [3, 5, 10]

    slope_window:
      value: 5
      robustness_neighbors: [3, 5, 10]
```

注意：

> 这些邻域参数不用于挑选“最赚钱”的版本，而用于判断结论是否只存在于单点。

合理结果应该是附近参数方向相近，而不是只有5日赚钱、4日和6日都崩溃。

---

## 3. 收益集中度

对于Top2尤其重要。

必须输出：

```yaml
concentration:
  top_trades:
    - 10
    - 20
  top_stocks:
    - 10
    - 20
  top_days:
    - 10
    - 20
```

并计算：

* 最佳10笔交易贡献比例
* 最佳10只股票贡献比例
* 最佳10个交易日贡献比例
* 去掉最佳10笔后的收益
* 去掉最差10笔后的收益

---

## 4. Regime暂时不要默认全展开

你的平台未来会有很多Regime。

如果每个因子默认按所有Regime切片，很快就会变成过拟合机器。

正确做法是：

```yaml
regime_analysis:
  mode: diagnostic
  min_samples_per_bucket: 100
  max_regimes_per_experiment: 3
```

Regime结果默认只用于解释，不用于自动判定因子有效。

只有事先声明经济假设：

```text
该因子预计在行业扩散加速环境有效
```

才允许把该Regime作为正式验证条件。

---

# 七、实验空间如何控制

推荐用阶段性矩阵，而不是总笛卡尔积。

## L1矩阵

假设：

* 5个预测周期
* 2个股票池
* 2个版本

总共：

```text
5 × 2 × 2 = 20组
```

这是可以接受的。

## L2矩阵

假设只让L1通过的最佳方向和合理股票池进入：

* 4个TopN
* 5个持有期
* 3个成本情景

理论上60组，但0bps和10bps可以基于同一笔交易记录重新计算，不需要重新跑撮合。

实际上只需要：

```text
4 × 5 = 20个核心组合
```

## L3

只对少数候选因子运行。

因此，绝大多数无效因子会在L1停止。

# 八、实验展开规则需要避免“自动选最佳”

系统可以排名，但不能把最佳结果直接宣布为有效。

例如系统测试：

```text
5个周期 × 4个TopN × 2个股票池
```

最后发现：

```text
Top2 + 3D + liquid
```

年化最高。

这只能说明它是本轮最佳结果，不能说明它是样本外可用参数。

系统必须记录：

```yaml
selection_metadata:
  total_combinations_tested: 40
  selected_by: exploratory_ranking
  requires_oos_confirmation: true
```

否则会隐藏多重试验问题。

# 十、我认为这里还有三个关键决策要明确

## 1. `Top2`是核心目标还是诊断项

结合你现在实际选股方式，我建议：

```text
Top2是核心结果
Top5/10/20是稳健性曲线
```

而不是四个TopN同等地挑最优。

---

## 2. 股票池是否固定

如果股票池每次跟着因子变化，你的结果很难比较。

建议：

* `tradeable`固定为广义研究池
* `liquid`固定为实盘池
* 个别因子需要特殊股票池时显式声明例外，但不能默默修改

---

## 3. 是否允许自动搜索参数

我的建议是：

```text
允许批量跑参数邻域
不允许自动把最佳参数当正式结论
```

机器可以发现候选，但必须在冻结参数后的独立样本上重新验证。

---

这套规则的核心不是让系统“尽可能多地跑”，而是让它做到：

> 无效因子尽快淘汰；有线索的因子进入可交易性验证；只有真正有希望的因子才运行完整稳健性测试。

Alpha评分标准不应该只是：

> 年化越高，分越高。

否则系统会自动偏爱高换手、高集中、过拟合的Top2策略。

更合理的结构是：

> **硬性门槛 + 分维度评分 + 最终分级**

也就是说，先判断有没有明显致命问题，再给候选因子打分。

---

# 一、先明确评分对象

评分对象不是某一个最优参数组合，而应该是：

> 一个因子在固定实验模板下的整体表现。

例如一个因子会产生：

* raw / neutralized
* tradeable / liquid
* 1 / 3 / 5 / 10 / 15日
* Top2 / 5 / 10 / 20

不能选出其中最好看的一个结果直接评分。

评分时应该区分：

```text
主结果
辅助结果
稳健性结果
```

结合你的使用方式，我建议：

```text
主结果：
liquid 股票池
Top2
20bps成本
T+1开盘交易
```

辅助结果：

```text
Top5 / Top10 / Top20
其他持有期
tradeable股票池
```

所以评分核心仍围绕你真正会交易的Top2，而不是完全按照传统IC研究设计。

---

# 二、评分前先设硬性否决项

以下情况不管收益多高，都不能进入有效因子库。

## 1. 数据问题

```yaml
hard_reject:
  future_data_violation: true
  coverage_below_minimum: true
  invalid_price_usage: true
  survivorship_bias_detected: true
```

一旦存在未来数据、幸存者偏差、成交价格错误，直接：

```text
INVALID
```

不是低分，而是实验无效。

---

## 2. 不可交易

例如：

* 大量信号股票涨停无法买入
* 停牌股票仍被选入
* 跌停时假设能够卖出
* 成本后收益完全消失
* 策略容量远低于实际资金需求

这种状态不能叫Alpha通过。

---

## 3. 样本外彻底反向

例如：

```text
样本内年化 +30%
样本外年化 -15%
样本外IC方向相反
```

这种不应依靠其他维度加分救回来。

建议直接标记：

```text
REJECT_OOS
```

---

## 4. 收益完全由极少数交易贡献

例如：

```text
最佳10笔交易贡献 > 100%
去掉最佳10笔后策略亏损
```

不一定立刻永久否决，但至少不能进入正式候选，应标记：

```text
REVIEW_CONCENTRATION
```

---

# 三、评分维度建议

我建议总分100分，分成六个维度。

| 维度       |  权重 |
| -------- | --: |
| 样本外有效性   |  25 |
| 可交易收益    |  25 |
| 稳定性      |  20 |
| TopN结构   |  10 |
| 统计证据     |  10 |
| 独立性与可解释性 |  10 |
| 合计       | 100 |

这个权重是针对你这种：

* A股
* 短线
* Top2集中选股
* 持仓1～15日

而设计的。

传统机构可能把IC和统计显著性权重放得更高，但你不能因为全截面IC一般，就直接否定头部型信号。

---

# 四、样本外有效性：25分

这是最高优先级之一。

## 评分内容

### 1. OOS收益方向

```text
OOS成本后超额收益 > 0
```

### 2. OOS收益保留率

```text
OOS收益 / IS收益
```

不是越高越好，而是不能明显崩塌。

例如：

| OOS保留率  | 得分 |
| ------- | -: |
| ≥70%    |  8 |
| 40%～70% |  6 |
| 20%～40% |  3 |
| 0～20%   |  1 |
| <0      |  0 |

### 3. OOS IC方向

如果因子是头部型，可以允许IC较弱，但方向不能持续相反。

### 4. Walk-forward一致性

例如5个滚动测试窗口中：

* 4个为正：高分
* 3个为正：中等
* 1～2个为正：低分

建议拆分：

```yaml
oos_score:
  net_excess_return: 8
  retention_ratio: 6
  walk_forward_consistency: 7
  oos_direction_consistency: 4
```

---

# 五、可交易收益：25分

这是你的第二核心。

主要看主策略：

```text
liquid + Top2 + 20bps + 实际交易约束
```

## 评分内容

### 1. 成本后超额收益

注意必须是：

```text
策略收益 - 同股票池等权基准
```

不能只看绝对年化。

例如：

| 成本后年化超额 | 得分 |
| ------- | -: |
| ≥15%    | 10 |
| 10%～15% |  8 |
| 5%～10%  |  6 |
| 2%～5%   |  3 |
| 0～2%    |  1 |
| ≤0      |  0 |

这个阈值以后可以根据你的平台总体结果调整。

### 2. Calmar

Top2集中策略不能只看Sharpe。

建议：

| Calmar  | 得分 |
| ------- | -: |
| ≥1.0    |  6 |
| 0.7～1.0 |  5 |
| 0.4～0.7 |  3 |
| 0.2～0.4 |  1 |
| <0.2    |  0 |

### 3. 成本敏感度

比较：

```text
0bps
10bps
20bps
```

如果20bps后仍能保留大部分Alpha，说明可交易性更强。

### 4. 交易实现率

例如实际可买入信号占比：

```text
executed_signals / generated_signals
```

涨停买不到的比例过高，需要扣分。

建议：

```yaml
tradability_score:
  annualized_excess_return: 10
  calmar: 6
  cost_resilience: 5
  execution_feasibility: 4
```

---

# 六、稳定性：20分

稳定性不是要求每年赚钱，而是避免依赖单一时期。

## 1. 年度正收益比例

例如10年中：

| 正超额年份比例 | 得分 |
| ------- | -: |
| ≥70%    |  6 |
| 60%～70% |  5 |
| 50%～60% |  3 |
| 40%～50% |  1 |
| <40%    |  0 |

## 2. 滚动窗口稳定

例如滚动12个月、24个月超额收益为正的比例。

## 3. 参数邻域稳定

例如主参数：

```text
return_window = 5
slope_window = 5
```

邻域：

```text
3 / 5 / 10
```

如果只有5有效，其他全失效，说明单点过拟合。

## 4. 收益集中度

检查：

* 最佳10笔交易贡献
* 最佳10只股票贡献
* 最佳10个交易日贡献
* 去掉最佳10笔后是否仍为正

建议：

```yaml
stability_score:
  year_consistency: 6
  rolling_consistency: 5
  parameter_neighborhood: 5
  contribution_concentration: 4
```

---

# 七、TopN结构：10分

这一项是专门为你的Top2策略加的。

你需要判断因子属于：

```text
全截面型
头部断层型
偶然极端型
```

## 理想情况

```text
Top2 > Top5 > Top10 > Top20 > 基准
```

不要求完全平滑，但应该存在可解释的衰减。

### 评分建议

| 表现                           | 得分 |
| ---------------------------- | -: |
| Top2最好，Top5/10/20仍保持正超额且有序衰减 | 10 |
| Top2和Top5有效，Top10后接近基准       |  7 |
| 只有Top2有效，但跨年度稳定              |  5 |
| 只有Top2有效且依赖少数交易              |  2 |
| Top2不稳定或TopN完全无规律            |  0 |

这一项不能只判断“单调性”。

因为你的信号可能本来就是头部型。

---

# 八、统计证据：10分

这里不要权重过高，但不能没有。

## 指标

* Rank IC
* ICIR
* IC正值比例
* FDR q
* 分组收益
* 单调性
* OOS IC

建议不要采用单一阈值，例如：

```text
IC必须大于0.03
```

因为不同股票池、持有期和因子类型差异很大。

建议综合评分：

```yaml
statistical_score:
  ic_strength: 3
  ic_stability: 3
  fdr_significance: 2
  quantile_structure: 2
```

对于头部型因子，`quantile_structure`可以替换为：

```text
extreme_tail_significance
```

也就是专门检验Top2、Top5是否显著高于其他股票。

---

# 九、独立性与可解释性：10分

这是防止因子库里堆满重复因子。

## 1. 与已有因子相关性

例如新因子与现有五日动量相关系数达到0.95，即使能赚钱，也可能没有增量价值。

建议：

| 与现有最相近因子相关性 | 得分 |
| ----------- | -: |
| <0.30       |  4 |
| 0.30～0.50   |  3 |
| 0.50～0.70   |  2 |
| 0.70～0.85   |  1 |
| >0.85       |  0 |

## 2. 增量价值

将新因子加入已有组合后：

* 是否提升OOS收益
* 是否提升Calmar
* 是否降低回撤
* 是否改善覆盖范围

## 3. 经济解释

因子是否有明确机制：

```text
为什么会产生收益？
为什么尚未被完全套利？
什么情况下会失效？
```

经济解释不能证明因子有效，但没有解释的因子更难长期维护。

建议：

```yaml
independence_score:
  low_factor_correlation: 4
  incremental_portfolio_value: 4
  economic_explanation: 2
```

---

# 十、最终评分不要只有一个数字

建议系统同时输出：

```yaml
alpha_assessment:
  total_score: 72

  dimension_scores:
    oos: 18
    tradability: 20
    stability: 14
    topn_structure: 8
    statistics: 5
    independence: 7

  hard_flags:
    future_data: false
    oos_reversal: false
    excessive_concentration: true
    cost_failure: false

  classification: REVIEW
```

因为两个同样是72分的因子，结构可能完全不同。

一个可能是：

* 收益高
* 统计弱
* 集中度高

另一个可能是：

* 收益中等
* 很稳定
* OOS好
* 容量高

不能只看总分。

# 十一、建议的因子分级

## INVALID

存在数据泄漏、幸存者偏差、回测错误。

## REJECTED

```text
总分 < 50
```

或者触发关键硬性否决。

没有足够Alpha证据，不继续投入。

## WATCHLIST

```text
50～64
```

有一些线索，但不足以进入正式候选。

适合：

* 调整因子机制
* 检查作用域
* 探索条件化
* 暂不进入组合

## CANDIDATE

```text
65～79
```

存在可交易Alpha线索，进入更严格样本外或仿真。

## APPROVED

```text
≥80
```

且必须同时满足：

```text
OOS维度 ≥ 17/25
可交易维度 ≥ 17/25
稳定性维度 ≥ 12/20
没有关键硬性风险
```

不能只靠某一项特别高，把总分抬到80。

## PRODUCTION

不建议仅靠回测评分自动进入。

它还应经过：

* 冻结参数
* 独立测试集
* 模拟盘
* 实际成交验证
* 容量评估
* 风险预算

# 十二、两类因子需要不同子评分

这是一个重要设计。

## 1. 全截面型因子

特点：

* IC稳定
* 分组收益单调
* TopN逐步衰减

它更看重：

```text
IC
ICIR
单调性
中性化后表现
```

## 2. 头部型因子

特点：

* Top2、Top5明显突出
* 全截面IC可能不高
* 中间股票没有稳定顺序

它更看重：

```text
Top2 OOS收益
TopN断层结构
收益集中度
极端尾部显著性
交易可实现性
```

建议因子输入里新增：

```yaml
factor:
  expected_shape: monotonic | top_tail | bottom_tail | unknown
```

解释：

* `monotonic`：预期全截面单调
* `top_tail`：预期仅头部有效
* `bottom_tail`：预期底部极端有效
* `unknown`：探索阶段由系统诊断，但正式验证前冻结

这一字段很有价值，因为不能拿同一套标准强行评估所有因子。

# 十四、我的核心建议

你的系统最终不要输出：

```text
这个因子有Alpha：是 / 否
```

应该输出：

```text
因子类型：头部型
总分：72
状态：CANDIDATE
优势：Top2成本后OOS收益稳定
缺陷：全截面排序弱、收益集中度偏高
下一步：进入冻结参数后的独立测试集
```

这样系统才能真正帮助你淘汰因子，而不是自动制造“高分幻觉”。

这套Alpha评分标准的核心是：

> 收益重要，但样本外、可交易、稳定性必须共同成立；任何一项严重缺失，都不能靠总分掩盖。

可以开发，但还不能直接开工写完整平台。

你现在已经定好了三块最核心的“研究协议”：

1. 因子输入格式：因子怎么描述
2. 实验展开规则：因子要经过哪些测试
3. Alpha评分标准：结果如何淘汰和晋级

这三块解决了过去最大的问题：

> 每次都临时定义因子、临时写回测、临时解释结果。

但它们目前只定义了**研究流程的上层规则**。要成为真正可复用的平台，还缺少几份底层契约。否则开发过程中仍然会不断临时决定，最后又变成一次性系统。

## 还缺的五个关键部分

### 一、数据契约

因子配置已经声明了：

```yaml
required_fields:
  - close
  - volume
  - industry
```

但系统还没有明确这些字段到底是什么。

例如 `close` 必须定义：

* 原始价、前复权还是后复权
* 停牌日如何处理
* 涨跌停价格如何处理
* 除权除息日收益如何计算
* 数据什么时候可被使用
* 缺失值是否前向填充

`industry` 也必须定义：

* 使用申万几级行业
* 历史行业归属还是当前行业归属
* 行业变更何时生效
* 一只股票无行业时如何处理

你需要一份统一的数据字典，例如：

```yaml
field: close
frequency: daily
adjustment: backward_adjusted
available_time: trade_date_close
nullable: false
source_priority:
  - tushare
```

如果这层不固定，同一个因子在不同阶段可能算出不同结果。

---

### 二、公式 DSL 的精确定义

你现在有这种公式：

```yaml
formula: slope(cs_rank(ret(close, 5), by=industry), window=5)
```

但必须继续规定每个算子的严格语义。

例如：

```text
ret(close, 5)
```

到底是：

```text
close_t / close_t-5 - 1
```

还是五个单日收益的累乘？

```text
cs_rank
```

输出是：

* 1到N名次
* 0到1分位数
* -1到1标准化排名

并列值怎么处理？

```text
slope
```

是：

* 首尾差
* OLS回归斜率
* 是否除以均值
* 是否需要至少5个有效点

这些不能交给每个开发者自行理解。

你至少需要一份：

> 算子注册表 + 参数规范 + 输入输出类型 + 空值规则

例如：

```yaml
operator: slope
input_type: time_series
parameters:
  window:
    type: integer
    min: 2
method: ols
min_periods: window
output: regression_coefficient
```

这份契约非常关键。

---

### 三、回测执行契约

实验模板写了：

```yaml
signal_time: close_t
entry_price: open_t1
holding_periods: [1, 3, 5, 10, 15]
overlapping_portfolios: true
```

但执行引擎还要明确更多细节。

例如Top2持有5天时：

* 每天买入新的2只，是否形成10个子仓位
* 每批资金分配比例是多少
* 老持仓再次进入Top2时，是加仓还是延长持有
* 涨停买不到时，空仓还是递补Top3
* 跌停卖不出时，是否继续持有
* 停牌期间收益如何计入
* 现金收益是否计算
* 组合净值在开盘还是收盘更新

这些不是实现细节，而是会直接改变回测结果。

建议先固定一个简单而唯一的第一版：

```text
每日生成一批独立子组合
每批投入总资金的1/holding_period
TopN等权
买不到不递补
卖不出则顺延至可卖日
同一股票可被多个子组合重复持有
```

或者采用其他规则，但必须写死并做版本管理。

---

### 四、实验产物与存储契约

平台不能只生成一份Markdown报告。

每次实验必须能被完整复现，因此至少要保存：

```text
因子配置原文
实验配置原文
评分配置原文
数据版本
代码版本
运行时间
随机种子
因子值
每日持仓
交易明细
指标结果
错误日志
最终评分
```

建议每次运行都有唯一 `run_id`：

```text
sector_rank_slope__20260630__a81f2c
```

并保存一份不可修改的 manifest：

```yaml
run_id: ...
factor_version: ...
experiment_profile: ...
scoring_profile: ...
data_snapshot: ...
git_commit: ...
status: success
```

否则三个月后你无法回答：

> 为什么上次年化20%，这次只有8%？

---

### 五、自动测试和防作弊规则

这是量化平台里最容易被忽略的一层。

平台必须自动测试：

* 因子值是否使用未来数据
* 收益标签是否错位
* T日收盘信号是否错误使用T日收盘成交
* 历史股票池是否存在幸存者偏差
* 财务数据是否按照真实公告日期进入
* 复权数据是否与成交价格混用
* 行业归属是否使用了未来分类
* 涨停、停牌是否正常拦截
* 相同输入是否产生相同结果

至少要有几组人工可计算的小样本测试。

例如用3只股票、10天数据，手工确定：

```text
ret_5d
行业排名
Top2
持有期
成本
组合净值
```

然后让自动化测试逐项核对。

没有这一层，批量回测只会更快地批量制造错误结果。

## 所以现在能做到什么程度

你已经可以开始开发平台的**MVP骨架**，但不要直接实现所有因子和所有报告。

第一版目标应该限定为：

> 给定一个日频因子YAML，不修改代码，系统能够计算因子、执行固定实验、生成评分和保存完整产物。

建议第一版只支持：

* 日频数据
* A股
* 价格、成交量、市值、行业
* 有限DSL算子
* `market`和`industry`两类横截面
* Top2/5/10/20
* 1/3/5/10/15日
* 固定交易模型
* 三层实验和评分输出

暂时不要支持：

* 分钟数据
* 逐笔成交
* 自由Python插件
* 机器学习模型
* 任意策略逻辑
* 自动参数优化
* 多数据源动态切换
* 复杂组合优化器

这些会让平台再次失控。

## 我建议的完整平台边界

可以把系统固定成六层：

```text
1. 数据层
   标准字段、历史快照、可得时间

2. 因子DSL层
   解析YAML并生成因子值

3. 因子质量层
   L0检查

4. 研究评估层
   IC、分组收益、OOS、衰减

5. 组合回测层
   TopN、持有期、成本、交易约束

6. 评分与产物层
   Alpha评分、分类、报告、结果存储
```

OpenClaw更适合放在最外层：

```text
接收研究想法
→ 生成标准因子YAML
→ 调用平台CLI
→ 读取标准结果
→ 决定淘汰或进入下一阶段
```

不要让OpenClaw自己临时写回测代码。它应该调用稳定平台，而不是成为平台本身。

## 判断平台是否真正可复用

最终用下面这四个问题验收：

1. 新增因子是否只需要新增YAML？
2. 同一个因子能否用不同实验模板运行，而不修改因子代码？
3. 同一个结果是否能够根据 `run_id` 完整复现？
4. 无效因子是否能自动在L0或L1停止，而不是跑完整流程？

四个都能做到，才算真正的批量因子研究平台。

因此目前的判断是：

> 三个模板已经足以确定平台的上层骨架，但正式开发前还必须定下数据契约、DSL算子契约、回测执行契约、实验产物契约和测试规则。

下一步最应该定的是**数据契约**。因为公式、回测和评分最终都依赖数据口径；数据口径不稳定，后面的系统越自动化，错误传播得越快。

可以。数据契约我会直接采用适合 **A股日频、Top2、持仓1—15日、批量因子研究** 的默认方案，只把确实依赖你现状或资金规模的问题留下。

## 我直接确定的默认方案

| 项目          | 默认决定                                         |
| ----------- | -------------------------------------------- |
| 市场范围        | 沪深主板、创业板、科创板；第一版排除北交所、ETF、可转债                |
| 数据频率        | 仅支持日频                                        |
| 历史区间        | 2015-01-01 至最新完整交易日                          |
| 主数据源        | Tushare                                      |
| 数据读取        | 回测只读取本地已发布数据版本，不在回测中临时调用接口                   |
| 股票标识        | `ts_code`                                    |
| 历史股票        | 保留退市股票，按历史上市、退市状态生成股票池                       |
| 新股限制        | 上市满60个交易日后才可进入选股池                            |
| 时间标准        | 交易日、`Asia/Shanghai`                          |
| 原始价格        | 永久保存未复权 OHLC，用于成交、涨跌停和资产估值                   |
| 因子价格        | 使用 `close × adj_factor` 构造稳定连续价格序列           |
| 收益计算        | 使用复权因子调整后的连续收益，包含分红送转影响                      |
| 成交价格        | 只允许使用未复权原始价格                                 |
| 成交量单位       | 股                                            |
| 成交额单位       | 元                                            |
| 收益单位        | 小数，5%存储为 `0.05`                              |
| 停牌          | 当日不可选；持仓按最后收盘价估值；无法卖出则顺延                     |
| 涨跌停         | 按板块和历史交易规则计算；开盘涨停不能买，开盘跌停不能卖                 |
| 买不到         | 不递补，资金保留为现金                                  |
| ST          | 使用历史时点ST状态，ST及*ST全部排除                        |
| 市值中性化       | 默认使用 `log(total_mv)`；同时保留流通市值                |
| 行业体系        | 默认申万一级行业                                     |
| 行业数据        | 必须使用历史时点行业归属；当前行业回填历史只能用于调试                  |
| 行业最小样本      | 有效股票少于10只，整组因子设为无效，不回退全市场                    |
| 股票池层级       | 区分因子计算池、选股池、实际可买池                            |
| `tradeable` | 非ST、非停牌、非退市整理、上市满60日                         |
| `liquid`    | `tradeable`基础上，20日平均成交额≥5000万元，20日至少成交18日    |
| 缺失行情        | 不填充，不用0替代                                    |
| 财务数据        | 未来扩展时严格按真实公告可得时间，不按报告期使用                     |
| 因子缺失        | 横截面排名时排除，不进行0填充                              |

## 二、存储架构

你接受 SQLite 和 Parquet，不使用 DuckDB，因此这样分工。

### SQLite：控制面和小型维表


作用分别是：

* 管理数据契约版本；
* 登记每个数据集的来源和字段；

* 记录每次拉取任务；

## 四、行情与复权口径

不直接保存一个所谓“前复权价格”作为事实数据。

保存：

```text
raw_open
raw_high
raw_low
raw_close
pre_close
adj_factor
```

运行时或数据加工层生成：

```text
adj_open  = raw_open  × adj_factor
adj_high  = raw_high  × adj_factor
adj_low   = raw_low   × adj_factor
adj_close = raw_close × adj_factor
```

使用规则：

| 场景         | 使用价格      |
| ---------- | --------- |
| 因子计算       | 复权连续价格    |
| IC未来收益标签   | 复权连续价格    |
| 组合持有收益     | 复权连续价格    |
| 涨跌停判断      | 原始价格和涨跌停价 |
| 买入、卖出成交价记录 | 原始价格      |
| 成交额、滑点和容量  | 原始成交额     |

这样既不会把虚构的复权价格当成实际成交价格，也不会在除权除息日制造错误收益。

Tushare的前复权结果会受到查询 `end_date` 影响，所以平台不能依赖每次调用 `pro_bar(qfq)` 得到的结果作为不可变数据源。平台保存原始行情和复权因子，自己构造统一连续价格更容易复现。([Tushare][4])

## 五、行业数据规则

默认使用：

```yaml
industry_standard: SW2021
industry_level: L1
point_in_time_required: true
min_group_size: 10
```

Tushare当前提供 `index_classify` 和 `index_member_all` 两类申万行业数据接口，可用于分类和成分建设。([Tushare][5])

必须将行业成分整理成区间表：

```text
ts_code
industry_code
industry_name
in_date
out_date
source_version
```

某日股票的行业归属只能按：

```text
in_date <= trade_date < out_date
```

确定。

以下情况需要自动处理：

* 如果账号没有申万接口权限：行业因子禁止正式运行。
* 如果2015年以来历史行业覆盖不完整：允许开发调试，但评分状态必须为 `INVALID_DATA_COVERAGE`。
* 禁止使用当前行业归属回填全部历史。
* 某行业有效股票不足10只：该行业当日因子值为空，不回退到全市场排序。

## 六、股票池规则

### 因子计算池

```text
当日已经上市
尚未退市
上市满60个交易日
有有效因子所需数据
```

停牌股票是否进入因子计算池，由具体算子决定；但默认横截面排名排除停牌股票。

### `tradeable`选股池

```text
因子计算池
- ST和*ST
- 停牌
- 退市整理期
```

### `liquid`实盘研究池

因为你的资金规模在100万元以内，采用：

```yaml
liquidity:
  avg_amount_window: 20
  min_avg_amount_cny: 30000000
  min_traded_days: 18
  max_estimated_participation_rate: 0.02
```

也就是：

* 过去20日平均成交额至少3000万元；
* 过去20日至少交易18日；
* 单次预计成交金额不超过股票日均成交额的2%。

这个门槛对100万元以内的Top2组合已经足够保守，没有必要一开始固定到5000万元。

## 九、权限不足的处理

因为你现在只有 Token，不确定积分权限，初始化阶段必须自动检测：

```yaml
required_api_check:
  - stock_basic
  - trade_cal
  - daily
  - adj_factor
  - daily_basic
  - stk_limit
  - suspend
  - st_status
  - index_classify
  - index_member_all
  - index_daily
```

结果分三类：

```text
AVAILABLE
PERMISSION_DENIED
INSUFFICIENT_HISTORY
```

其中：

* 缺少行业数据，只禁用行业类因子；
* 缺少 `daily_basic`，禁用市值中性化和 `liquid`池正式评分；
* 缺少日线或复权因子，整个数据版本禁止发布；
* 不能静默降级后继续给出Alpha结论。

至此，数据契约的关键决策已经全部确定。下一步应该把这些内容固化为 `data_contract_v1.yaml`、`field_dictionary_v1.yaml` 和 `data_quality_rules_v1.yaml`，再进入DSL算子契约。
